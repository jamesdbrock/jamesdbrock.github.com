4 ## Introduction
5 
6 The High Frequency FIX Parser library is an open source implementation of <a href="http://www.fixtradingcommunity.org/pg/structure/tech-specs/fix-protocol">Financial Information Exchange protocol versions 4.2, 4.3, 4.4, and 5.0 SP2.</a> intended for use by developers of high frequency, low latency financial software. The purpose of the library is to do fast, efficient encoding and decoding of FIX in place, at the location of the I/O buffer. The library does not use intermediate message objects, and it does **no memory allocation** on the free store (the &ldquo;heap&rdquo;).
7 
8 Hffix library is not certified by any industry-leading committees. It is not an &ldquo;engine.&rdquo; It is not an &ldquo;adaptor.&rdquo; It has no threading, no I/O, no object-oriented inheritance. It is just a superfast parser and serializer in plain modern generic-iterator-style C++03.

42 ## Library Design
43 
44 High Frequency FIX Parser tries to follow the
45 <a href="http://www.boost.org/development/requirements.html">Boost Library Requirements and Guidelines</a>. It is modern platform-independent C++03 and depends only on the C++ Standard Library. It is patterned after the C++ Standard Template Library, and it models each FIX message with Container and Iterator concepts. It employs compile-time generic templates but does not employ object-oriented inheritance.
46 
47 High Frequency FIX Parser is a header-only library, so there are no binaries to link. It also plays well with Boost. If you are using <a href="http://www.boost.org/doc/html/date_time.html">Boost Date_Time</a> in your application, High Frequency FIX Parser will support conversion between FIX fields and Boost Date_Time types.
48 
49 All of the Financial Information Exchange (FIX) protocol specification versions supported by the library are bundled into the the distribution, in the `spec` directory. As a convenience for the developer, the High Frequency FIX Parser library includes a Python script which parses all of the FIX protocol specification documents and generates the `include/hffix_fields.hpp` file. That file has `enum` definitions in a tag namspace and an `hffix::dictionary_init_field` function which allows fields to be referred to by name instead of number both during development, and at run-time.
50 
51 The design criteria for High Frequency FIX Parser are based on our experience passing messages to various FIX hosts for high frequency quantitative trading at <a href="http://www.t3live.com">T3 Trading Group, LLC</a>.
52

67 ## Features
68 
69 ### Other FIX Implementations
70 
71 Typical FIX implementations employ object-oriented-style programming to model a FIX message either as an associative key-value container of strongly typed objects that inherit from some field superclass, or as a class type for each message type with member variables for every possible field in the message.
72 
73 There are two disadvantages to this method.
74 
75 1. Creating these message objects requires free-store memory allocation, which uses a lot of CPU time &mdash; typically more CPU time than all the rest of the parsing logic.
76 2. Declaring the classes for these objects requires a lot of boilerplate code, and makes it difficult to handle surprising messages at run-time.
77 
78 The advantage of object-oriented-style FIX parsers is that with the familiar object API, any field of a message object can be read or written randomly at any point in the program, which may simplify program logic.
79 
80 ### High Frequency FIX Parser Implementation
81 
82 For reading FIX messages, High Frequency FIX Parser presents an STL-style <a href="http://www.sgi.com/tech/stl/ForwardIterator.html">immutable Forward Iterator</a> interface. Writing fields is done serially with an interface similar to an STL-style <a href="http://www.sgi.com/tech/stl/BackInsertionSequence.html">Back Insertion Sequence Container</a>. Reading and writing are done directly on the I/O buffer, without any intermediate objects.
83 
84 The disadvantage of this implementation is that the message API provides serial access to fields, not random access.
85 
86 The advantage is that this enables the High Frequency FIX Parser library to completely avoid free store memory allocation.
87 The library performs all memory allocation on the stack, and the library never requires developers using the library to allocate anything on the free store with `new` or `malloc`.
88 
89 Field values in the FIX protocol are always encoded on the wire as ASCII, and High Frequency FIX Parser exposes field values to the developer as iterator range `char const* begin(), char const* end()`. High Frequency FIX Parser also provides a complete set of conversion functions to native C++ types for *ints*, *decimal floats*, *dates* and *times*, et cetera &mdash; see documentation for `hffix::message_writer` and `hffix::field_value`.
90 
91 ### Exceptions
92 
93 All High Frequency FIX Parser methods, functions, constructors, and destructors provide the No-Throw exception guarantee unless they are documented to throw exceptions, in which case they provide the Basic exception guarantee. Generally, a function may throw `std::logic_error` if preconditions are not met by the programmer. See documentation for details.
94 
95 ### Thread Safety
96 
97 High Frequency FIX Parser is not thread-aware at all and has no threads, mutexes, locks, or atomic operations.
98 
99 All `const` methods of the `hffix::message_reader` are safe for concurrent calls.
100 
101 The `hffix::message_writer` is not safe for concurrent calls.
102 
103 `hffix::message_reader` and `hffix::message_writer` have no storage of their own, they read and write fields directly on an I/O buffer. The developer must guarantee that the buffer endures while fields are being read or written.
104 
105 ### FIX Sessions
106 
107 Managing sessions requires making choices about sockets and threads. High Frequency FIX Parser does not manage sessions. It is intended for developers who want a FIX parser with which to build a session manager for a high-performance trading system that already has a socket and threading architecture.
108 
109 FIX has transport-layer features mixed in with the messages, and most FIX hosts have various quirks in the way they employ the administrative messages. To manage a FIX session your application will need to match the the transport-layer and administrative features of the other FIX host. High Frequency FIX Parser has the flexibility to express any subset or proprietary superset of FIX.
110 
111 Consult [FIX Session-level Test Cases and Expected Behaviors](http://www.fixtradingcommunity.org/pg/file/fplpo/read/30489/fix-sessionlevel-test-cases-and-expected-behaviors)
112 
113 ### Encryption
114 
115 High Frequency FIX Parser supports the binary data field types such as *SecureData*, but it does not implement any of the *EncryptMethods* suggested by the FIX specifications. If you want to encrypt or decrypt some data you'll have to do the encryption or decryption yourself.
116 
117 ### CheckSum
118 
119 High Frequency FIX Parser can calculate CheckSums and add the *CheckSum* field for all messages that you encode. It does not validate the *CheckSum* of messages decoded.
120 
121 ### Sequence Numbers
122 
123 The *MsgSeqNum* field in the FIX Standard Header is exposed for reading and writing.
124 
125 ### Administrative Messages
126 
127 The administrative messages *Logon*, *Logout*, *ResendRequest*, *Heartbeat*, *TestRequest*, *SeqReset-Reset* and *SeqReset-GapFill* don't get special treatment in High Frequency FIX Parser. Any administrative message can be encoded or decoded like any other message.
128 
129 ### User-Defined Fields and Custom Tags
130 
131 High Frequency FIX Parser does not enforce the data type of the Field Definitions for content fields in the FIX spec, so the developer is free to read or write any tag number with any field data type.

138 ### Writing a Message Example
139 
140 This example program is in the _hffix_ repository at `test/src/writer01.cpp`.
141 
142 It writes a _Logon_ message and a _New Order - Single_ message to `stdout`.
143 
144 ~~~cpp
145 #include <iostream>
146 
147 // We want Boost Date_Time support, so include these before hffix.hpp.
148 #include <boost/date_time/posix_time/posix_time_types.hpp>
149 #include <boost/date_time/gregorian/gregorian_types.hpp>
150 
151 #include <hffix.hpp>
152 
153 int main(int argc, char** argv)
154 {
155  int sequence_number_send(0);
156 
157  char buffer[1 << 13];
158 
159  // We'll put a FIX Logon message in the buffer.
160  hffix::message_writer logon(buffer, buffer + sizeof(buffer));
161 
162  logon.push_back_header("FIX.4.2"); // Write BeginString and BodyLength.
163 
164  logon.push_back_string (hffix::tag::MsgType, "A"); // Logon MsgType.
165  logon.push_back_string (hffix::tag::SenderCompID, "AAAA"); // Required Standard Header field.
166  logon.push_back_string (hffix::tag::TargetCompID, "BBBB"); // Required Standard Header field.
167  logon.push_back_int (hffix::tag::MsgSeqNum, ++sequence_number_send); // Required Standard Header field.
168  logon.push_back_timestamp (hffix::tag::SendingTime,
169  boost::posix_time::microsec_clock::universal_time()); // Required Standard Header field.
170  logon.push_back_int (hffix::tag::EncryptMethod, 0); // no encryption
171  logon.push_back_int (hffix::tag::HeartBtInt, 10); // 10 second heartbeat interval
172 
173  logon.push_back_trailer(); // write CheckSum
174 
175  // Now the Logon message is written to the buffer.
176 
177  // Add a FIX New Order - Single message to the buffer, after the Logon message.
178  hffix::message_writer new_order(logon.message_end(), buffer + sizeof(buffer));
179 
180  new_order.push_back_header("FIX.4.2");
181 
182  new_order.push_back_string (hffix::tag::MsgType, "D"); // New Order - Single
183  new_order.push_back_string (hffix::tag::SenderCompID, "AAAA"); // Required Standard Header field.
184  new_order.push_back_string (hffix::tag::TargetCompID, "BBBB"); // Required Standard Header field.
185  new_order.push_back_int (hffix::tag::MsgSeqNum, ++sequence_number_send); // Required Standard Header field.
186  new_order.push_back_timestamp (hffix::tag::SendingTime,
187  boost::posix_time::microsec_clock::universal_time()); // Required Standard Header field.
188  new_order.push_back_string (hffix::tag::ClOrdID, "A1");
189  new_order.push_back_char (hffix::tag::HandlInst, '1'); // Automated execution.
190  new_order.push_back_string (hffix::tag::Symbol, "OIH");
191  new_order.push_back_char (hffix::tag::Side, '1'); // Buy.
192  new_order.push_back_timestamp (hffix::tag::TransactTime,
193  boost::posix_time::microsec_clock::universal_time());
194  new_order.push_back_int (hffix::tag::OrderQty, 100); // 100 shares.
195  new_order.push_back_char (hffix::tag::OrdType, '2'); // Limit order.
196 
197  // Limit price $500.01 = 50001*(10^-2). The push_back_decimal() method takes a decimal floating point
198  // number of the form mantissa*(10^exponent).
199  new_order.push_back_decimal (hffix::tag::Price, 50001, -2);
200 
201  new_order.push_back_char (hffix::tag::TimeInForce, '1'); // Good Till Cancel.
202 
203  new_order.push_back_trailer();
204 
205  //Now the New Order message is in the buffer after the Logon message.
206 
207  // Write both messages to stdout.
208  std::cout.write(buffer, new_order.message_end() - buffer);
209 
210  return 0;
211 }
212 ~~~

215 ### Reading a Message Example
216 
217 This example program is in the _hffix_ repository at `test/src/reader01.cpp`.
218 
219 It reads messages from `stdin`. If it finds a _Logon_ message or a _New Order - Single_ message, then it prints out some information about their fields.
220 
221 ~~~cpp
222 #include <iostream>
223 #include <cstdio>
224 #include <map>
225 
226 // We want Boost Date_Time support, so include these before hffix.hpp.
227 #include <boost/date_time/posix_time/posix_time.hpp>
228 #include <boost/date_time/gregorian/gregorian.hpp>
229 
230 #include <hffix.hpp>
231 
232 const size_t chunksize = 4096; // Choose a preferred I/O chunk size.
233 
234 char buffer[1 << 20]; // Must be larger than the largest FIX message size.
235 
236 int main(int argc, char** argv)
237 {
238  std::map<int, std::string> field_dictionary;
239  hffix::dictionary_init_field(field_dictionary);
240 
241  size_t buffer_length(0); // The number of bytes read in buffer[].
242 
243  size_t fred; // Number of bytes read from fread().
244 
245  // Read chunks from stdin until 0 is read or the buffer fills up without finding a complete message.
246  while((fred = std::fread(buffer + buffer_length, 1, std::min(sizeof(buffer) - buffer_length, chunksize), stdin))) {
247 
248  buffer_length += fred;
249  hffix::message_reader reader(buffer, buffer + buffer_length);
250 
251  // Try to read as many complete messages as there are in the buffer.
252  for (; reader.is_complete(); reader = reader.next_message_reader()) {
253  if (reader.is_valid()) {
254 
255  // Here is a complete message. Read fields out of the reader.
256  try {
257  if (reader.message_type()->value() == "A") { // Logon message
258  std::cout << "Logon message\n";
259 
260  hffix::message_reader::const_iterator i = reader.begin();
261 
262  if (reader.find_with_hint(hffix::tag::SenderCompID, i))
263  std::cout << "SenderCompID = " << i++->value() << '\n';
264 
265  if (reader.find_with_hint(hffix::tag::MsgSeqNum, i))
266  std::cout << "MsgSeqNum = " << i++->value().as_int<int>() << '\n';
267 
268  if (reader.find_with_hint(hffix::tag::SendingTime, i))
269  std::cout << "SendingTime = " << i++->value().as_timestamp() << '\n';
270 
271  std::cout << "The next field is " << hffix::field_name(i->tag(), field_dictionary) << " = " << i->value() << '\n';
272 
273  std::cout << '\n';
274  }
275  else if (reader.message_type()->value() == "D") { // New Order Single message
276  std::cout << "New Order Single message\n";
277 
278  hffix::message_reader::const_iterator i = reader.begin();
279 
280  if (reader.find_with_hint(hffix::tag::Side, i))
281  std::cout << (i++->value().as_char() == '1' ? "Buy " : "Sell ");
282 
283  if (reader.find_with_hint(hffix::tag::Symbol, i))
284  std::cout << i++->value() << " ";
285 
286  if (reader.find_with_hint(hffix::tag::OrderQty, i))
287  std::cout << i++->value().as_int<int>();
288 
289  if (reader.find_with_hint(hffix::tag::Price, i)) {
290  int mantissa, exponent;
291  i->value().as_decimal(mantissa, exponent);
292  std::cout << " @ $" << mantissa << "E" << exponent;
293  ++i;
294  }
295 
296  std::cout << "\n\n";
297  }
298 
299  } catch(std::exception& ex) {
300  std::cerr << "Error reading fields: " << ex.what() << '\n';
301  }
302 
303  } else {
304  // An invalid, corrupted FIX message. Do not try to read fields out of this reader.
305  // The beginning of the invalid message is at location reader.message_begin() in the buffer,
306  // but the end of the invalid message is unknown (because it's invalid).
307  // Stay in this for loop, because the messager_reader::next_message_reader() function
308  // will see that this message is invalid and it will search the remainder of the buffer
309  // for the text "8=FIX", to see if there might be a complete or partial valid message
310  // anywhere else in the remainder of the buffer.
311  std::cerr << "Error Corrupt FIX message: ";
312  std::cerr.write(reader.message_begin(), std::min(ssize_t(64), buffer + buffer_length - reader.message_begin()));
313  std::cerr << "...\n";
314  }
315  }
316  buffer_length = reader.buffer_end() - reader.buffer_begin();
317 
318  if (buffer_length > 0) // Then there is an incomplete message at the end of the buffer.
319  std::memmove(buffer, reader.buffer_begin(), buffer_length); // Move the partial portion of the incomplete message to buffer[0].
320  }
321 
322  return 0;
323 }
324 
325 ~~~

393 ### Multi-threaded Sending
394 
395 Q: I have a bunch of different threads serializing and sending FIX messages out one socket. When each message is sent it needs a *MsgSeqNum*, but at serialization time I don't know what the *MsgSeqNum* will be, I only know that at sending time.
396 
397 A: That multi-threading model is not a good choice for your software. The performance penalty for that threading model is much greater than the performance advantage of this non-allocating parser library. You should consider redesigning to use a single-threaded simultaneous-wait event loop like *libev* or *Boost Asio*. If you insist on multi-threading, then you could do something like this code example.
398 
399 ~~~cpp
400 hffix::message_writer m;
401 m.push_back_string(hffix::tag::MsgSeqNum, "00000000"); // Make a placeholder value over which you can later paste your sequence number.
402 
403 // This thread_safe_send() function will correctly sequence FIX messages if two threads are racing to call thread_safe_send().
404 void thread_safe_send(hffix::message_writer const& w) {
405  lock l(send_mutex_); // Serialize access to this function.
406  hffix::message_reader r(w); // Construct a reader from the writer.
407  hffix::message_reader::const_iterator i = std::find(r.begin(), r.end(), hffix::tag_equal(hffix::tag::MsgSeqNum)); // Find the MsgSeqNum field.
408  if (i != r.end()) {
409  std::snprintf(const_cast<char*>(i->begin()), i->size(), "%.8i", next_sequence_number++); // Overwrite the "00000000" string with the next_sequence_number.
410  write(fd, w.message_begin(), w.message_size()); // Send the message to the socket.
411  }
412 }
413 ~~~

415 ### FIX Repeating Groups
416 
417 From *FIX-50_SP2_VOL-1_w_Errata_20110818.pdf* page 21:
418 
419 <blockquote>If the repeating group is used, the first field of the repeating group is required. This allows
420 implementations of the protocol to use the first field as a "delimiter" indicating a new repeating group
421 entry. The first field listed after the NoXXX, then becomes conditionally required if the NoXXX field
422 is greater than zero.</blockquote>
423 
424 The beginning of each Repeating Group is marked by a field with a &ldquo;NoXXX&rdquo; field. By convention, Repeating Groups are usually located at the end of the message, so the end of the message marks the end of the Repeating Group. In this example we assume that the convention holds, and the repeating group is at the end of the message. If the repeating group were not at the end of the message then we'd have to pay attention to the value of the &ldquo;NoXXX&ldquo; fields, which is left as an exercise for the reader.
425 
426 This is an example of iterating over the nested Repeating Groups when reading a *Mass Quote* message.
427 The *Mass Quote* message has *QuoteSet* Repeating Groups, and nested inside those groups are *QuoteEntry* Repeating Groups, see *fix-42-with_errata_20010501.pdf* page 52.
428 In each repeated *QuoteSet* Group, `hffix::tag::QuoteSetID` is always the first field. In each repeated *QuoteEntry* Group, `hffix::tag::QuoteEntryID` is always the first field.
429

481 ## Notes on the Design of FIX Protocol
482 
483 ### The *Logon* - *Resend Request* Race Condition
484 
485 When a FIX client connects to a FIX server, the client doesn't know what sequence number to use for the *Logon* message.
486 
487 Either the client can choose to reset both sequence numbers, in which case the client may miss messages, or not, in which case the client is subject to the *Resend Request* race condition.
488 
489 After *Logon* response from the server, the client may begin sending messages, but the client has to wait some amount of time because the server may send *Resend Request*. If the client sends any message to the server while the server is preparing to send *Resend Request*, then the server's response is not defined by the *FIX* specification, and some servers implementations may seize up in confusion at that point.

493 ## C++14
494 
495 This library only depends on C++03 because it doesn't need any of the features of C++14. It was, however, written with the intention of being included in a C++14 build, and will interact well with all C++14 features such as, for example, `auto`, or anonymous inline functions passed as the `UnaryPredicate` to `hffix::find_with_hint`.
