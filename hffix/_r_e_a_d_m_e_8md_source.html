6 ## Introduction
7 
8 The High Frequency FIX Parser library is an open source implementation of <a href="http://www.fixtradingcommunity.org/pg/structure/tech-specs/fix-protocol">Financial Information Exchange protocol versions 4.2, 4.3, 4.4, and 5.0 SP2.</a> intended for use by developers of high frequency, low latency financial software. The purpose of the library is to do fast, efficient encoding and decoding of FIX in place, at the location of the I/O buffer. The library does not use intermediate message objects, and it does **no memory allocation** on the free store (the “heap”).
9 
10 *hffix* library is not certified by any industry-leading committees. It is not an “engine.” It is not an “adaptor.” It has no threading, no I/O, no object-oriented subtyping. It is just a superfast parser and serializer in plain modern generic-iterator-style C++98.

44 ## Library Design
45 
46 High Frequency FIX Parser tries to follow the
47 <a href="https://github.com/isocpp/CppCoreGuidelines/blob/master/CppCoreGuidelines.md">C++ Core Guidelines</a> and the
48 <a href="http://www.boost.org/development/requirements.html">Boost Library Requirements and Guidelines</a>. It is modern platform-independent header-only C++98 and depends only on the C++ Standard Library. It is patterned after the C++ Standard Template Library, and it models each FIX message with Container and Iterator concepts. It employs compile-time generic templates but does not employ object-oriented inheritance.
49 
50 ### Speed
51 
52 The design criteria for High Frequency FIX Parser are based on our experience passing messages to various FIX hosts for high frequency quantitative trading at <a href="http://www.t3live.com">T3 Trading Group, LLC</a>. These design criteria follow from the observation that the latency of a trading system depends on the following list of considerations, in descending order of importance.
53 
54 1. __Network architecture.__ Most of the latency will be in the network between us and our peer, outside of our control. We should do everything we can to exert some control over the network to shorten our path.
55 2. __Operating System I/O syscall usage.__ We should be very careful about which syscalls we're using to do network I/O. Perhaps use an OS bypass NIC.
56 3. __Operating System thread usage.__ We should be running a single pinned operating system thread per core with a multiplexed event loop like *Boost Asio* or *libuv* for waiting on I/O events.
57 4. __Userspace runtime memory usage.__ Memory allocations can trigger a page fault and a syscall. Complicated object-oriented designs with lots of pointer indirections lead to cache misses and branch prediction failures.
58 5. __Userspace algorithms.__ This is the easiest stuff to measure and the most fun to discuss, so a lot of attention is focused here, but it's only important if we get all the other considerations right first.
59 
60 The *hffix* library assumes that the library user will want to make their own choices about considerations __2__ and __3__, and the *hffix* library focuses on providing good answers for consideration __4__. It does this by using only stack memory and I/O buffer memory, and never allocating on the free store.
61 
62 In contrast, the popular alternative *QuickFix* library forces the user to use the *QuickFix* solution to considerations __2__ and __3__ for threads and sockets, and most of *QuickFix*'s choices about threads and sockets are not great. *QuickFix* also has an inefficient object-oriented design for consideration __4__.
63 
64 See also <a href="https://www.youtube.com/watch?v=NH1Tta7purM">CppCon 2017: Carl Cook “When a Microsecond Is an Eternity: High Performance Trading Systems in C++”</a>
65 
66 ### Specs Included
67 
68 All of the Financial Information Exchange (FIX) protocol specification versions supported by the library are bundled into the the distribution, in the `spec` directory. As a convenience for the developer, the High Frequency FIX Parser library includes a Python script which parses all of the FIX protocol specification documents and generates the `include/hffix_fields.hpp` file. That file has `enum` definitions in a tag namspace and an `hffix::dictionary_init_field` function which allows fields to be referred to by name instead of number during both compile-time and run-time.

87 ## Features
88 
89 ### Serial Message Field Access
90 
91 For reading FIX messages, High Frequency FIX Parser presents an STL-style
92 <a href="http://en.cppreference.com/w/cpp/concept/ForwardIterator">immutable Forward Iterator</a>
93 interface. Writing fields is done serially with an interface similar to an STL-style
94 <a href="http://en.cppreference.com/w/cpp/concept/SequenceContainer">Back Insertion Sequence Container</a>.
95 Reading and writing are done directly on the I/O buffer, without any intermediate objects.
96 
97 The disadvantage of this implementation is that the message API provides serial access to fields, not random access. Of course, when we're writing a message, random access isn't important, just write out the fields in order. When we're reading a message, it's easy enough to pretend that we have random access by using iterator algorithms like `std::find`. A convenience algorithm `hffix::message_reader::find_with_hint` is provided by this library for efficiently reading fields when you know approximately what field order to expect. See the examples below for how this works out in practice.
98 
99 The advantage is that this enables the High Frequency FIX Parser library to completely avoid free store memory allocation.
100 The library performs all memory allocation on the stack, and the library never requires developers using the library to allocate anything on the free store with `new` or `malloc`.
101 
102 Field values in the FIX protocol are always encoded on the wire as ASCII, and High Frequency FIX Parser exposes field values to the developer as iterator range `char const* begin(), char const* end()`. High Frequency FIX Parser also provides a complete set of conversion functions to native C++ types for *ints*, *decimal floats*, *dates* and *times*, et cetera — see documentation for `hffix::message_writer` and `hffix::field_value`.
103 
104 ### Exceptions
105 
106 Some functions in this library may throw `std::logic_error` if a precondition is not met by the programmer, so you can usually prevent the library from throwing exceptions by meeting the precondition. All methods, functions, constructors, and destructors provide the No-Throw exception guarantee unless they are documented to throw exceptions, in which case they provide the Basic exception guarantee. See documentation for details.
107 
108 ### Thread Safety
109 
110 High Frequency FIX Parser is not thread-aware at all and has no threads, mutexes, locks, or atomic operations.
111 
112 All `const` methods of the `hffix::message_reader` are safe for concurrent calls.
113 
114 The `hffix::message_writer` is not safe for concurrent calls.
115 
116 `hffix::message_reader` and `hffix::message_writer` have no storage of their own, they read and write fields directly on an I/O buffer. The developer must guarantee that the buffer endures while fields are being read or written.
117 
118 ### FIX Sessions
119 
120 Managing sessions requires making choices about sockets and threads. High Frequency FIX Parser does not manage sessions. It is intended for developers who want a FIX parser with which to build a session manager for a high-performance trading system that already has a socket and threading architecture.
121 
122 FIX has transport-layer features mixed in with the messages, and most FIX hosts have various quirks in the way they employ the administrative messages. To manage a FIX session your application will need to match the the transport-layer and administrative features of the other FIX host. High Frequency FIX Parser has the flexibility to express any subset or proprietary superset of FIX.
123 
124 See also [FIX Session-level Test Cases and Expected Behaviors](http://www.fixtradingcommunity.org/pg/file/fplpo/read/30489/fix-sessionlevel-test-cases-and-expected-behaviors)
125 
126 ### Numerics
127 
128 No native floating-point numeric types (`double`, `float`) are employed by the library.
129 ASCII-encoded decimal numbers are represented by integral mantissa and exponent.
130 See `hffix::message_writer::push_back_decimal()` and `hffix::field_value::as_decimal()`.
131 As with every FIX data type, the High Frequency FIX library user has the option to serialize
132 and deserialize numeric fields themself rather than use these methods.
133 
134 ### Encryption
135 
136 High Frequency FIX Parser supports the binary data field types such as *SecureData*, but it does not implement any of the *EncryptMethods* suggested by the FIX specifications. If you want to encrypt or decrypt some data you'll have to do the encryption or decryption yourself.
137 
138 ### CheckSum
139 
140 High Frequency FIX Parser can calculate CheckSums and add the *CheckSum* field for all messages that you encode. It does not validate the *CheckSum* of messages decoded.
141 
142 ### Sequence Numbers
143 
144 The *MsgSeqNum* field in the FIX Standard Header is exposed for reading and writing.
145 
146 ### Administrative Messages
147 
148 The administrative messages *Logon*, *Logout*, *ResendRequest*, *Heartbeat*, *TestRequest*, *SeqReset-Reset* and *SeqReset-GapFill* don't get special treatment in High Frequency FIX Parser. Any administrative message can be encoded or decoded like any other message.
149 
150 ### User-Defined Fields and Custom Tags
151 
152 High Frequency FIX Parser does not enforce the data type of the Field Definitions for content fields in the FIX spec, so the developer is free to read or write any tag number with any field data type. See `hffix::message_writer` and `hffix::field_value` documentation under Extension for details.

155 ## Using High Frequency FIX Parser
156 
157 ### Writing a Message Example
158 
159 This example program is in the *hffix* repository at `test/src/writer01.cpp`.
160 
161 It writes a _Logon_ message and a _New Order - Single_ message to `stdout`.
162 
163 ~~~cpp
164 // We want Boost Date_Time support, so include these before hffix.hpp.
165 #include <boost/date_time/posix_time/posix_time_types.hpp>
166 #include <boost/date_time/gregorian/gregorian_types.hpp>
167 
168 #include <hffix.hpp>
169 #include <iostream>
170 
171 using namespace boost::posix_time;
172 using namespace boost::gregorian;
173 
174 int main(int argc, char** argv)
175 {
176  int seq_send(1); // Sending sequence number.
177 
178  char buffer[1 << 13];
179 
180  ptime tsend(date(2017,8,9), time_duration(12,34,56));
181 
182  // We'll put a FIX Logon message in the buffer.
183  hffix::message_writer logon(buffer, buffer + sizeof(buffer));
184 
185  logon.push_back_header("FIX.4.2"); // Write BeginString and BodyLength.
186 
187  // Logon MsgType.
188  logon.push_back_string (hffix::tag::MsgType, "A");
189  logon.push_back_string (hffix::tag::SenderCompID, "AAAA");
190  logon.push_back_string (hffix::tag::TargetCompID, "BBBB");
191  logon.push_back_int (hffix::tag::MsgSeqNum, seq_send++);
192  logon.push_back_timestamp (hffix::tag::SendingTime, tsend);
193  // No encryption.
194  logon.push_back_int (hffix::tag::EncryptMethod, 0);
195  // 10 second heartbeat interval.
196  logon.push_back_int (hffix::tag::HeartBtInt, 10);
197 
198  logon.push_back_trailer(); // write CheckSum.
199 
200  // Now the Logon message is written to the buffer.
201 
202  // Add a FIX New Order - Single message to the buffer, after the Logon
203  // message.
204  hffix::message_writer new_order(logon.message_end(), buffer + sizeof(buffer));
205 
206  new_order.push_back_header("FIX.4.2");
207 
208  // New Order - Single
209  new_order.push_back_string (hffix::tag::MsgType, "D");
210  // Required Standard Header field.
211  new_order.push_back_string (hffix::tag::SenderCompID, "AAAA");
212  new_order.push_back_string (hffix::tag::TargetCompID, "BBBB");
213  new_order.push_back_int (hffix::tag::MsgSeqNum, seq_send++);
214  new_order.push_back_timestamp (hffix::tag::SendingTime, tsend);
215  new_order.push_back_string (hffix::tag::ClOrdID, "A1");
216  // Automated execution.
217  new_order.push_back_char (hffix::tag::HandlInst, '1');
218  // Ticker symbol OIH.
219  new_order.push_back_string (hffix::tag::Symbol, "OIH");
220  // Buy side.
221  new_order.push_back_char (hffix::tag::Side, '1');
222  new_order.push_back_timestamp (hffix::tag::TransactTime, tsend);
223  // 100 shares.
224  new_order.push_back_int (hffix::tag::OrderQty, 100);
225  // Limit order.
226  new_order.push_back_char (hffix::tag::OrdType, '2');
227  // Limit price $500.01 = 50001*(10^-2). The push_back_decimal() method
228  // takes a decimal floating point number of the form mantissa*(10^exponent).
229  new_order.push_back_decimal (hffix::tag::Price, 50001, -2);
230  // Good Till Cancel.
231  new_order.push_back_char (hffix::tag::TimeInForce, '1');
232 
233  new_order.push_back_trailer(); // write CheckSum.
234 
235  //Now the New Order message is in the buffer after the Logon message.
236 
237  // Write both messages to stdout.
238  std::cout.write(buffer, new_order.message_end() - buffer);
239 
240  return 0;
241 }
242 ~~~

245 ### Reading a Message Example
246 
247 This example program is in the *hffix* repository at `test/src/reader01.cpp`.
248 
249 It reads messages from `stdin`. If it finds a _Logon_ message or a _New Order - Single_ message, then it prints out some information about their fields.
250 
251 ~~~cpp
252 #include <iostream>
253 #include <cstdio>
254 #include <map>
255 
256 // We want Boost Date_Time support, so include these before hffix.hpp.
257 #include <boost/date_time/posix_time/posix_time.hpp>
258 #include <boost/date_time/gregorian/gregorian.hpp>
259 
260 #include <hffix.hpp>
261 
262 const size_t chunksize = 4096; // Choose a preferred I/O chunk size.
263 
264 char buffer[1 << 20]; // Must be larger than the largest FIX message size.
265 
266 int main(int argc, char** argv)
267 {
268  int return_code = 0;
269 
270  std::map<int, std::string> field_dictionary;
271  hffix::dictionary_init_field(field_dictionary);
272 
273  size_t buffer_length(0); // The number of bytes read in buffer[].
274 
275  size_t fred; // Number of bytes read from fread().
276 
277  // Read chunks from stdin until 0 is read or the buffer fills up without
278  // finding a complete message.
279  while ((fred = std::fread(
280  buffer + buffer_length,
281  1,
282  std::min(sizeof(buffer) - buffer_length, chunksize),
283  stdin
284  )
285  )) {
286 
287  buffer_length += fred;
288  hffix::message_reader reader(buffer, buffer + buffer_length);
289 
290  // Try to read as many complete messages as there are in the buffer.
291  for (; reader.is_complete(); reader = reader.next_message_reader()) {
292  if (reader.is_valid()) {
293 
294  // Here is a complete message. Read fields out of the reader.
295  try {
296  if (reader.message_type()->value() == "A") {
297  std::cout << "Logon message\n";
298 
299  hffix::message_reader::const_iterator i = reader.begin();
300 
301  if (reader.find_with_hint(hffix::tag::SenderCompID, i))
302  std::cout
303  << "SenderCompID = "
304  << i++->value() << '\n';
305 
306  if (reader.find_with_hint(hffix::tag::MsgSeqNum, i))
307  std::cout
308  << "MsgSeqNum = "
309  << i++->value().as_int<int>() << '\n';
310 
311  if (reader.find_with_hint(hffix::tag::SendingTime, i))
312  std::cout
313  << "SendingTime = "
314  << i++->value().as_timestamp() << '\n';
315 
316  std::cout
317  << "The next field is "
318  << hffix::field_name(i->tag(), field_dictionary)
319  << " = " << i->value() << '\n';
320 
321  std::cout << '\n';
322  }
323  else if (reader.message_type()->value() == "D") {
324  std::cout << "New Order Single message\n";
325 
326  hffix::message_reader::const_iterator i = reader.begin();
327 
328  if (reader.find_with_hint(hffix::tag::Side, i))
329  std::cout <<
330  (i++->value().as_char() == '1' ?"Buy ":"Sell ");
331 
332  if (reader.find_with_hint(hffix::tag::Symbol, i))
333  std::cout << i++->value() << " ";
334 
335  if (reader.find_with_hint(hffix::tag::OrderQty, i))
336  std::cout << i++->value().as_int<int>();
337 
338  if (reader.find_with_hint(hffix::tag::Price, i)) {
339  int mantissa, exponent;
340  i->value().as_decimal(mantissa, exponent);
341  std::cout << " @ $" << mantissa << "E" << exponent;
342  ++i;
343  }
344 
345  std::cout << "\n\n";
346  }
347 
348  } catch(std::exception& ex) {
349  std::cerr << "Error reading fields: " << ex.what() << '\n';
350  }
351 
352  } else {
353  // An invalid, corrupted FIX message. Do not try to read fields
354  // out of this reader. The beginning of the invalid message is
355  // at location reader.message_begin() in the buffer, but the
356  // end of the invalid message is unknown (because it's invalid).
357  //
358  // Stay in this for loop, because the
359  // messager_reader::next_message_reader() function will see
360  // that this message is invalid and it will search the
361  // remainder of the buffer for the text "8=FIX", to see if
362  // there might be a complete or partial valid message anywhere
363  // else in the remainder of the buffer.
364  //
365  // Set the return code non-zero to indicate that there was
366  // an invalid message, and print the first 64 chars of the
367  // invalid message.
368  return_code = 1;
369  std::cerr << "Error Invalid FIX message: ";
370  std::cerr.write(
371  reader.message_begin(),
372  std::min(
373  ssize_t(64),
374  buffer + buffer_length - reader.message_begin()
375  )
376  );
377  std::cerr << "...\n";
378  }
379  }
380  buffer_length = reader.buffer_end() - reader.buffer_begin();
381 
382  if (buffer_length > 0)
383  // Then there is an incomplete message at the end of the buffer.
384  // Move the partial portion of the incomplete message to buffer[0].
385  std::memmove(buffer, reader.buffer_begin(), buffer_length);
386  }
387 
388  return return_code;
389 }
390 ~~~

428 ## Boost Date_Time
429 
430 If the <a href="http://www.boost.org/doc/html/date_time.html">Boost Date_Time</a> library is available in your build environment, `boost::posix_time::ptime`, `boost::posix_time::time_duration`, and `boost::gregorian::date` will be automatically supported for the various FIX date and time field types.
431 
432 To enable High Frequency FIX Parser support for the Boost Date_Time library types, include the Boost libraries before the hffix.hpp library, like this:
433 
434 ~~~cpp
435 #include <boost/date_time/posix_time/posix_time_types.hpp>
436 #include <boost/date_time/gregorian/gregorian_types.hpp>
437 #include <hffix.hpp>
438 ~~~
439 
440 To prevent High Frequency FIX Parser support for the Boost Date_Time library, `#define HFFIX_NO_BOOST_DATETIME` before including `hffix.hpp`:
441 
442 ~~~cpp
443 #define HFFIX_NO_BOOST_DATETIME
444 #include <hffix.hpp>
445 ~~~

459 ### Multi-threaded Sending
460 
461 Q: I have a bunch of different threads serializing and sending FIX messages out one socket. When each message is sent it needs a *MsgSeqNum*, but at serialization time I don't know what the *MsgSeqNum* will be, I only know that at sending time.
462 
463 A: That multi-threading model is not a good choice for your software. The performance penalty for that threading model is much greater than the performance advantage of this non-allocating parser library. You should consider redesigning to use a single-threaded simultaneous-wait event loop like *libev* or *Boost Asio*. If you insist on multi-threading, then you could do something like this code example.

466 hffix::message_writer m;
467 m.push_back_string(hffix::tag::MsgSeqNum, "00000000"); // Make a placeholder value over which you can later paste your sequence number.
468 
469 // This thread_safe_send() function will correctly sequence FIX messages if two threads are racing to call thread_safe_send().
470 void thread_safe_send(hffix::message_writer const& w) {
471  lock l(send_mutex_); // Serialize access to this function.
472  hffix::message_reader r(w); // Construct a reader from the writer.
473  hffix::message_reader::const_iterator i = std::find(r.begin(), r.end(), hffix::tag_equal(hffix::tag::MsgSeqNum)); // Find the MsgSeqNum field.
474  if (i != r.end()) {
475  std::snprintf(const_cast<char*>(i->begin()), i->size(), "%.8i", next_sequence_number++); // Overwrite the "00000000" string with the next_sequence_number.
476  write(fd, w.message_begin(), w.message_size()); // Send the message to the socket.
477  }
478 }

481 ### FIX Repeating Groups
482 
483 From *FIX-50_SP2_VOL-1_w_Errata_20110818.pdf* page 21:
484 
485 <blockquote>If the repeating group is used, the first field of the repeating group is required. This allows
486 implementations of the protocol to use the first field as a "delimiter" indicating a new repeating group
487 entry. The first field listed after the NoXXX, then becomes conditionally required if the NoXXX field
488 is greater than zero.</blockquote>
489 
490 The beginning of each Repeating Group is marked by a field with a “NoXXX” field. By convention, Repeating Groups are usually located at the end of the message, so the end of the message marks the end of the Repeating Group. In this example we assume that the convention holds, and the repeating group is at the end of the message. If the repeating group were not at the end of the message then we'd have to pay attention to the value of the “NoXXX” fields, which is left as an exercise for the reader.
491 
492 This is an example of iterating over the nested Repeating Groups when reading a *Mass Quote* message.
493 The *Mass Quote* message has *QuoteSet* Repeating Groups, and nested inside those groups are *QuoteEntry* Repeating Groups, see *fix-42-with_errata_20010501.pdf* page 52.
494 In each repeated *QuoteSet* Group, `hffix::tag::QuoteSetID` is always the first field. In each repeated *QuoteEntry* Group, `hffix::tag::QuoteEntryID` is always the first field.
495 
496 ~~~cpp
497 hffix::message_reader r;
498 
499 hffix::message_reader::const_iterator group1_begin = std::find_if(r.begin(), r.end(), hffix::tag_equal(hffix::tag::QuoteSetID));
500 hffix::message_reader::const_iterator group1_end;
501 
502 for (; group1_begin != r.end(); group1_begin = group1_end) {
503  group1_end = std::find_if(group1_begin + 1, r.end(), hffix::tag_equal(hffix::tag::QuoteSetID));
504 
505  // This loop body will be entered once for each QuoteSet Repeating Group.
506  //
507  // group1_begin will point to the first field in the QuoteSet group, which is always hffix::tag::QuoteSetID.
508  // group1_end will point past-the-end of the QuoteSet group.
509 
510  hffix::message_reader::const_iterator group2_begin = std::find_if(group1_begin, group1_end, hffix::tag_equal(hffix::tag::QuoteEntryID));
511  hffix::message_reader::const_iterator group2_end;
512 
513  for (; group2_begin != group1_end; group2_begin = group2_end) {
514  group2_end = std::find_if(group2_begin + 1, group1_end, hffix::tag_equal(hffix::tag::QuoteEntryID));
515 
516  // This loop body will be entered once for each QuoteEntry Repeating Group.
517  //
518  // group2_begin will point to the first field of the QuoteEntry group, which is always QuoteEntryID.
519  // group2_end will point past-the-end of the QuoteEntry group.
520  }
521 }
522 ~~~

539 ## Notes on the Design of FIX Protocol
540 
541 ### The *Logon* - *Resend Request* Race Condition
542 
543 When a FIX client connects to a FIX server, the client doesn't know what sequence number to use for the *Logon* message.
544 
545 Either the client can choose to reset both sequence numbers, in which case the client may miss messages, or not, in which case the client is subject to the *Resend Request* race condition.
546 
547 After *Logon* response from the server, the client may begin sending messages, but the client has to wait some amount of time because the server may send *Resend Request*. If the client sends any message to the server while the server is preparing to send *Resend Request*, then the server's response is not defined by the *FIX* specification, and some servers implementations may seize up in confusion at that point.
